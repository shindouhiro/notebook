# 工厂模式 (Factory Pattern)

工厂模式是一种创建型设计模式，它提供了一种创建对象的最佳方式。在工厂模式中，我们在创建对象时不会对客户端暴露创建逻辑，而是通过使用一个共同的接口来指向新创建的对象。

简单来说，**工厂模式就是将对象的创建过程封装起来**。

---

## 1. 核心概念
- **封装性**：将创建对象的逻辑放在一个中心位置。
- **解耦**：使用者只需要知道自己想要什么（参数），而不需要知道对象是如何被构造出来的。

---

## 2. JavaScript 实现方式

### 方式一：简单工厂模式 (Simple Factory)
简单工厂模式通过一个特定的函数或类来根据传入的参数决定创建哪一种产品类的实例。

```javascript
class Dog {
  speak() { console.log("Woof!"); }
}

class Cat {
  speak() { console.log("Meow!"); }
}

// 简单工厂
class AnimalFactory {
  static createAnimal(type) {
    switch (type) {
      case 'dog':
        return new Dog();
      case 'cat':
        return new Cat();
      default:
        throw new Error('Unknown animal type');
    }
  }
}

const myDog = AnimalFactory.createAnimal('dog');
myDog.speak(); // Woof!
```

### 方式二：工厂方法模式 (Factory Method)
工厂方法模式不再由一个单一的工厂类决定创建哪种产品，而是将具体产品的创建延迟到子类中。

```javascript
// 抽象工厂类
class UIProvider {
  createButton() {
    throw new Error("Method 'createButton()' must be implemented.");
  }
}

// 具体工厂 1: Windows 风格
class WindowsUI extends UIProvider {
  createButton() {
    return { style: "WinButton", label: "OK" };
  }
}

// 具体工厂 2: MacOS 风格
class MacOSUI extends UIProvider {
  createButton() {
    return { style: "MacButton", label: "Confirm" };
  }
}

const platform = "MacOS";
let factory;

if (platform === "Windows") {
  factory = new WindowsUI();
} else {
  factory = new MacOSUI();
}

const myBtn = factory.createButton();
```

## 3. 深度解析：为什么需要抛出异常的基类？

在示例代码中，我们定义了 `UIProvider` 并让其方法抛出异常。这在 JavaScript 中被称为 **“抽象基类 (Abstract Base Class)” 的模拟实现**。

### 它的作用：
1. **充当“接口 (Interface)”**：JS 原生不支持接口。通过基类抛错，我们实际上定义了一套**契约**：任何继承我的子类，**必须**实现这个方法，否则就报错。
2. **防止直接实例化**：我们不希望有人直接 `new UIProvider()`，因为它没有实际逻辑。抛错能强制开发者去写具体的实现类（如 `WindowsUI`）。
3. **接口一致性**：确保不管工厂怎么变，外部调用者永远只需要调用 `.createButton()`，保证了代码的稳定性。

---

## 4. 工厂方法模式 vs 简单工厂模式

### 核心区别：
- **简单工厂**：像是一个“全能店员”，所有的产品创建逻辑都塞在一个 `switch` 或 `if-else` 里。增加产品时得改动工厂内部逻辑。
- **工厂方法**：像是一个“品牌总代”，它只定义规格（基类），具体的生产交给各个品牌分厂（子类）。

### 为什么需要工厂方法？（优势）：
1. **符合“开闭原则 (Open-Closed Principle)”**：
   - 增加新产品时，**不需要修改已有的工厂代码**，只需新建一个子类即可。这极大地降低了引入新 Bug 的风险。
2. **解耦更彻底**：
   - 工厂类不再负责所有产品的逻辑。每个产品由自己的工厂子类负责，逻辑高度自治。
3. **支持复杂的对象构造**：
   - 如果不同产品的创建过程差异巨大，简单工厂会变得极其臃肿；工厂方法则可以把这些差异化逻辑完美分散到各个子类中。

---

## 5. 使用场景

### 1. 对象的创建逻辑比较复杂
如果一个对象的初始化需要大量的参数处理、环境判断或依赖注入，使用工厂模式可以把这些复杂性隐藏在工厂内部。

### 2. 需要根据不同环境创建不同实例
- **跨平台适配**：例如根据浏览器环境（Chrome/Firefox）或运行环境（Node/Browser）创建不同的网络请求模块。
- **UI 组件库**：根据配置生成不同样式的 Button、Input 等。

### 3. 处理大量重复类型的对象
例如在一个 RPG 游戏中，根据关卡难度批量产生不同等级的怪物。

### 4. 消除硬编码的依赖
工厂模式可以让你的代码不依赖于具体的类名，而是依赖于一个接口或参数，方便后续进行扩展（如增加新的产品类）。

---

## 4. 在前端开发中的应用案例

### `document.createElement`
这是 Web API 中最典型的简单工厂模式。你只需传入标签名（'div', 'span'），它就会返回对应的 DOM 对象，你不需要关心该对象内部复杂的构造过程。

```javascript
const div = document.createElement('div');
const img = document.createElement('img');
```

### 鉴权系统 (Auth Service)
根据用户选择的登录方式（微信、GitHub、账号密码），工厂返回不同的登录处理实例。

```javascript
class AuthFactory {
  getAuthMethod(provider) {
    if (provider === 'github') return new GitHubAuth();
    if (provider === 'wechat') return new WeChatAuth();
    // ...
  }
}
```

## 5. 优缺点分析

### 优点：
- **良好的封装性**：代码结构清晰，访问者无需了解实现细节。
- **极高的扩展性**：增加新产品时，只需在工厂中添加逻辑，不需要修改现有的客户端代码。
- **符合开闭原则**：对扩展开放，对修改关闭。

### 缺点：
- **代码复杂度增加**：每增加一个产品，都需要增加对应的类或逻辑，导致类数量增多。
- **工厂类责任重**：在简单工厂中，如果产品种类过多，工厂方法会变得非常臃肿。

---

> **总结**：当你发现自己在代码中到处都在 `new` 同一个基类的不同子类，或者对象的创建逻辑变得越来越难以维护时，就该考虑使用工厂模式了。
