# 单例模式 (Singleton Pattern)

单例模式是一种创建型设计模式，它确保一个类只有一个实例，并提供一个全局访问点来访问它。

在 JavaScript 中，单例模式非常常用，特别是在处理共享状态或全局资源时。

## 1. 核心概念
- **唯一性**：确保类只有一个实例。
- **全局访问**：该实例可以从应用程序的任何地方访问。

---

## 2. JavaScript 实现方式

### 方式一：ES6 Class (静态属性)
这是目前最直观、最常用的实现方式。

```javascript
class Singleton {
  constructor() {
    if (Singleton.instance) {
      return Singleton.instance;
    }
    this.state = {
      startTime: Date.now()
    };
    Singleton.instance = this;
  }

  static getInstance() {
    if (!Singleton.instance) {
      Singleton.instance = new Singleton();
    }
    return Singleton.instance;
  }
}

const instance1 = new Singleton();
const instance2 = new Singleton();

console.log(instance1 === instance2); // true
```

### 方式二：ES Modules (最推荐)
在现代 JavaScript 开发中，ES 模块天生就是单例的。当你导出一个实例时，无论在哪导入，它们引用的都是同一个对象。

```javascript
// counter.js
let count = 0;

const counter = {
  increment() {
    return ++count;
  },
  getCount() {
    return count;
  }
};

export default counter;

// main.js
import counter from './counter.js';
counter.increment();
```

### 方式三：闭包实现
利用闭包的私有作用域来保存实例。

```javascript
const Singleton = (function() {
  let instance;

  function createInstance() {
    return {
      message: "I am the lucky one"
    };
  }

  return {
    getInstance: function() {
      if (!instance) {
        instance = createInstance();
      }
      return instance;
    }
  };
})();

const s1 = Singleton.getInstance();
const s2 = Singleton.getInstance();
console.log(s1 === s2); // true
```

## 3. 使用场景

### 1. 全局状态管理 (State Management)
Redux 或 Vuex 的 Store 本质上就是一种单例。整个应用共享一个唯一的全局状态树，确保数据的一致性。

### 2. 数据库或外部服务连接 (Service Connection)
为了节省资源和管理连接，通常会使用单例模式来管理数据库连接池、WebSocket 或是第三方 API 的鉴权实例。

### 3. 配置对象 (Configuration)
应用的全局配置信息（如 API 环境地址、超时时间等）通常只需初始化一次，并在各处读取。

### 4. 浏览器端 UI 组件 (Global UI Components)
有些组件在屏幕上只能出现一个，例如：
- 全局加载提示 (Global Loading Spinner)
- 通知中心 (Notification Toast)
- 登录模态框 (Only one Login Modal should exist)

### 5. 缓存管理 (Cache)
用于实现应用内部的数据缓存，避免重复请求相同的 HTTP 接口。

## 4. 优缺点分析

### 优点：
- **资源优化**：避免重复创建，减少内存占用。
- **数据共享**：方便在不同模块间共享相同的实例数据。
- **严格控制**：确保访问的是同一个受控点。

### 缺点：
- **隐藏依赖**：由于是全局访问，会导致模块间的隐耦合，增加测试难度。
- **违反职责单一原则**：该类既负责实例的创建，又负责具体的业务逻辑。
- **扩展性差**：多线程环境下（JS虽是单线程，但 Worker 等场景下需注意）可能存在竞争（JS 相对较少）。

---

> **注意**：在 JavaScript 中，如果你的需求只是“存储一些全局变量”，通常直接使用对象字面量（Object Literal）或 ES 模块导出即可，没必要过度设计。
