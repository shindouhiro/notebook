# 🌟 @antfu/eslint-config 推荐

> [!abstract] 核心亮点
> - ✨ **扁平化配置** (Flat Config)
> - 🎨 **自带 ESLint 格式化**，完美替代 Prettier
> - 🚀 **开箱即用**的极致体验

前端大神 **Antfu**（Anthony Fu）等人认为 Prettier 的格式化规则过于死板（如部分强制换行策略），因此他们主导成立了 **[ESLint Stylistic](https://eslint.style/)** 项目，主张通过 ESLint 规则取代传统格式化工具，将主动排版权交还给开发者。

---

## 🤔 为什么逐渐放弃 Prettier？

在过去很长一段时间里，Prettier 是前端代码格式化的标配，但它也存在一些在大型项目或特定场景下令人头疼的问题：

### 1. 强制性的折行（The Maximum Line Length）
Prettier 最被诟病的一点是：一旦单行代码长度超过了定义的 `printWidth`（默认 80），它会自动、**无差别地**拆分所有的属性。
- 😠 **开发者的意图**：希望将几个高度相关的短属性或逻辑关联紧密的代码放在同一行，让代码结构更加紧凑直观。
- 🤖 **Prettier 的做法**：无视逻辑关联性，只要超过长度阈值就强制拆成每行一个属性，甚至有时候适得其反增加了视觉负担。

### 2. 空行的“强行支配”
为了区分不同的逻辑块，开发者往往会特意在代码间留出空行；反之，在某些紧凑逻辑中则希望避免产生多余空行。
- 🤖 **Prettier 的做法**：会武断地强制合并多出的空行，或者在它认为特定的语法层（如某些括号前后）强制删除或添加空行。代码失去了自由编排的“呼吸感”。
- 🌿 **ESLint Stylistic 的解法**：通过 `no-multiple-empty-lines` 等规则允许你配置合理的“最大连续空行数”，保留代码原有的“呼吸感”和节奏分布。

### 3. 对象和数组字面量的武断拆分
在处理长度较长的数组或对象时，Prettier 经常会做出违背直觉的排版决策。
- 🤖 **例子**：比如一个只包含 3 个简单元素的数组，可能因为其所在行的总长度恰巧超过了 `printWidth` 几个字符的位置，Prettier 就会果断把包含方括号在内的短短 3 个元素直接拆成 4 行，导致屏幕纵向空间被极大浪费。
- 💡 **Antfu 的逻辑**：如果开发者觉得这三个元素放一行依然清晰易读，那么格式化工具就不应该粗暴地强行介入拆开。

### 4. 注释的“异常位移”
Prettier 经常擅作主张地移动注释位置，这是非常危险的行为。
- 💥 **潜在影响**：如果你在某一行的末尾写了一段简短的补充注释，Prettier 可能会因为判定该行太长，强行把注释挤到下一行。这不仅打破了代码原有的信息排版，甚至改变了注释原本附着的代码行对象，导致阅读代码或 Review 时的理解偏差。

## 📦 内置对 pnpm Catalogs 的治理支持

除了代码格式化功能，该配置集极大地考量了现代 Monorepo 工程化的痛点，它内置并整合了对 **pnpm Catalogs** 的深度支持。

### 核心配置实现

```javascript
import antfu from '@antfu/eslint-config'

export default antfu({ 
   // 核心：开启 pnpm 治理规则集
   pnpm: true 
})
```

### 带来的工程化优势

> [!success] 统一依赖管理，告别版本混乱
> - 🔍 **自动对账**：底层引擎会自动定位分析整个 Monorepo 根目录下 `pnpm-workspace.yaml` 中的 `catalogs` 定义。
> - 🚫 **防劣化强制规范**：如果发现在任意子包的 `package.json` 里开发者习惯性地写死了具体依赖版本号（例如 `"axios": "^1.0.0"`），ESLint 会精准捕捉并将其标记为错误。
> - ⚡ **体验极佳的一键同步**：当开发者由于复制粘贴引入了不合规版本时，只需运行 `eslint --fix`，它会瞬间把全项目中所有对应依赖的版本号“一键扁平化”处理，统一归口修复为 `"axios": "catalog:"`。实现了依赖版本的**清晰分析**与**强制统一管理**。
